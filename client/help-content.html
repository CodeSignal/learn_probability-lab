Overview

CodeSignal Probability Lab is a simulation engine for repeated trials (coin, die, spinner, or custom devices) that helps you see probability stabilize: as you run more trials, the relative frequencies tend to settle near the theoretical probabilities.

You’ll use live charts and tables to connect:

  • Sample space (all possible outcomes)
  • Event (a subset of outcomes)
  • Theoretical probability (what “should” happen in the long run)
  • Experimental probability (what your data shows)

Competencies

  • Identifies probability as a measure of likelihood
  • Describes outcomes and sample spaces
  • Estimates event frequency from real or simulated data
  • Compares probabilities
  • Interprets relationships between events using tables or diagrams

Getting Started

  1. Choose a mode: One event (single sample space) or Two events (two-way table).
  2. Pick a device (coin / die / spinner / custom) and optionally adjust its bias.
  3. Define Event E by selecting outcomes (e.g., “Heads”, or “Even” by selecting 2, 4, 6).
  4. Run trials using Run, the +1/+10/+100/+1000 buttons, or Start auto.
  5. Interpret the results using the charts and tables (look at how the estimates stabilize as trials increase).

Key Features

Relative Frequency (Bar)

Shows the current relative frequency of each outcome. The more trials you run, the more stable the bars become.

Convergence (Event E)

Tracks your estimate of P(E) over time. Early results fluctuate a lot; larger sample sizes tend to “settle.”

Frequency Table

Lists theoretical probability, count, relative frequency, and the difference (Δ) between experiment and theory for each outcome. Selected outcomes (Event E) are highlighted.

Two Events: Heatmap + Two-way Table

Switch to Two events to explore joint outcomes. Click a cell to see:

  • P(A ∩ B) (joint probability estimate)
  • P(A | B) and P(B | A) (conditional probability estimate)
  • An "independence check" comparing P(A)·P(B) to P(A ∩ B)

Seeded Runs

Use a seed to repeat the same random sequence and compare runs.

Learning Workflow

Use these short tasks to build intuition quickly:

  1. Sample space: Choose a device and list its outcomes (e.g., die: 1–6).
  2. Event building: Define an event (e.g., “Even” on a die) by selecting outcomes.
  3. Convergence: Run 10, 100, 1,000 trials. Describe how the estimate changes.
  4. Comparing probabilities: Change the device bias (e.g., increase P(6)) and predict what will happen before running.
  5. Relationships: In two-event mode, compare Independent vs Dependent. Explain what changes in the two-way table.

Tips & Best Practices

  • Small samples are noisy on purpose—don’t “fix” the randomness.
  • If you change the device bias, treat it as a new experiment and reset.
  • Use the tables to justify claims (don’t rely only on what “feels” likely).

Shortcuts

  • ESC - Close help modal
  • Enter (in the Seed input) - Apply seed
  • Click a heatmap/two-way cell - Show joint + conditional probabilities

Troubleshooting / FAQ

Why doesn’t my experimental probability equal the theoretical probability?

Because randomness has variance. With small samples, fluctuations are expected. As you run more trials, relative frequencies tend to stabilize near the theoretical values.

What does “convergence” mean here?

It means the estimate (relative frequency) becomes more stable as the number of trials increases. It won’t be perfectly constant, but the wiggles usually shrink.

What’s the point of a “biased” coin/die/spinner?

To compare probabilities and see how changing the underlying distribution changes the long-run frequencies. Predict first, then test.

How does “Dependent” work with custom devices?

Dependent mode splits outcomes into a “low half” and “high half” by their order. If you want the dependence to feel meaningful, order outcomes from low → high.

What should I look for in the two-way table?

Start with the joint counts. Then click a cell to compare P(A ∩ B) to P(A)·P(B). If events are independent, these values should be close (with enough trials).

Is anything saved?

The lab saves basic settings (mode, device choices, bias sliders, event selection, seed) in your browser so you can continue where you left off. Trial results are not persisted.
